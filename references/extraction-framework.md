# 工程经验提炼框架 (Engineering Knowledge Extraction Framework)

> Cangjie 的 L1-L4 分层 schema。配套主入口 `../SKILL.md`(提炼流程)。
> worked example(§5)用 generic 示例,可直接公开。

从一次开发会话里,把混在一起的工程知识拆成 4 层,各归各位:
可迁移的进 skill 正文(越用越聪明),平台专属的进参数表(换平台只改数据),
会变的状态进 memory(本就该天天变)。

这是本方案的核心 —— 对标 nuwa 的「心智模型三重验证」,但提炼对象是**工程经验**而非人物思维。

---

## 0. 适用输入

提炼器读三样,合成"原材料":
- **会话 transcript** —— 任务是什么、试了什么、什么失败、什么成功、为什么这么定
- **git diff / commit** —— 实际改了什么
- **踩坑现场** —— 报错、回滚、卡住后怎么解的

输出:把原材料里每个"学习时刻"(一个决策 / 一次失败-修复 / 一个发现的事实)拆成 L1-L4 片段。

---

## 1. 核心判据:作用域(scope)

**唯一的尺子 = 这条知识能往多远迁移。** 作用域越宽 → 层越高 → 越该进 skill 正文(谁装都受益);
作用域越窄 → 层越低 → 越该外置成数据(换平台只换数据,不动 skill)。

| 层 | 判据(作用域) | 是什么 | 落点 |
|---|---|---|---|
| **L1 原理/约束** | 任何平台、任何 vendor 都成立 —— 是**问题本身的客观约束**,不是你的选择 | 物理/密码/协议不变量 | `SKILL.md` 正文 |
| **L2 设计/流程/坑** | 换平台仍成立,但**是你选的做法、你踩出的经验**(换个人可能用别的方法) | 设计模式、SOP、诊断手法、方法层的坑 | `SKILL.md` 正文 / `pitfalls.md` |
| **L3 平台参数/事实** | **换平台就变**的具体值/事实 | 地址、长度、密钥路径、工具名;**某版实现细节、上游结构事实、设计变更事实(含其 version-bound 的 why)** | `params/<平台>.md` |
| **L4 状态** | **下次会话就变** | 当前用哪把 key、当前 HEAD、哪块板 | `memory`,带时间戳 |

**关键澄清:稳定 ≠ 可迁移。** 一个具体密钥长度值对某颗芯片可能十年不变,但它是个**值**不是**理**,换芯片就可能变 → 还是 L3。
尺子是"作用域 + 是值还是理",不是"变不变"。

---

## 2. 路由测试(每条候选知识必跑,顺序不可乱)

```
Step 1 · 时效性筛
  这条下次会话/下次启动就会变吗?(当前 key / 当前 HEAD / 当前板号)
  └─ 是 → L4,结束
  └─ 否 → Step 2

Step 2 · 迁移性筛
  换一颗芯片 / 换一个平台,这条还成立吗?
  └─ 不成立(是具体值/地址/路径/某版实现细节)→ L3,结束
  └─ 成立 → Step 3

Step 3 · 是值还是理(排他性)
  这条,别人能不同意吗?
  └─ 不能不同意(问题的客观约束)→ L1
  └─ 能用别的方式(你的设计/经验/流程)→ L2

Step 4 · 拆检(对 L1/L2 结果强制执行)
  这条里面是否夹带了具体值 / 某平台实现细节?
  └─ 是 → 把夹带的部分切出来降到 L3,本条只留可迁移的骨架
  └─ 否 → 定稿
```

**失败分支(拿不准时):**
- 一条同时像 L1 又像 L3 → 它是没拆开的复合知识,回 Step 4 拆。
- L1 还是 L2 分不清 → **默认降到 L2**。见 §3 的不对称原则。

---

## 3. L1 vs L2:排他性测试 + 不对称原则

都"换平台成立",怎么分?用排他性:
- **L1 非排他** —— 所有做这件事的人都得认(OTP 不可逆,谁都改不了)。
- **L2 排他** —— 体现你的判断(信任链分几级、验签不过先查哪、SOP 步骤序)。换个工程师可能不这么干。

测试一句话:**"这条,别人能不同意吗?"** 不能 → L1;能 → L2。

**不对称原则(拿不准就往低放):**
放错方向的代价不对称 ——
- 把平台垃圾误塞进 L1 → **污染所有领域的 skill 正文**,谁装谁中招,最坏。
- 把好教训误压在 L3 → 顶多下次重新推一遍,**便宜**。

所以:**L1 ⟂ L2 拿不准 → 选 L2;L2 ⟂ L3 拿不准 → 选 L3。宁可窄,不可宽。**

---

## 4. 核心操作:「拆」,不是「贴一个标签」

这是本框架和 naive 打标签的本质区别。

**一条工程踩坑,几乎总是同时含「可迁移的教训」+「一次性的事实」,必须拆开存:**
- 教训 → L2(换平台还能救你)
- 事实 → L3(只对这平台成立)

不拆的后果:
- 教训被绑死在平台上 → 下次用不上(白踩)。
- 事实污染 skill 正文 → 换平台改不动(每次重写)。

**判据:一个知识片段,如果删掉所有平台专属的值/名字后还剩下有用的话 —— 剩下的是 L1/L2,删掉的是 L3。**

## 5. Worked Example —— 安全启动一级验签(generic)完整拆解

原始会话(混在一起的样子):

> "做某 SoC 的一级安全启动:公钥 hash 烧进片内 OTP,启动早期阶段读出来验一级镜像签名;签名用 RSA + PSS。
> 踩坑——从一个软件层 flag(启动参数里的字段)推断 OTP 没烧,错了,实际烧好了;真因是启动参数结构体里那个
> 公钥缓冲区按旧密钥长度定的固定大小,换更长的密钥后 memcpy 越界,把紧跟其后的字段冲成 0;
> 教训是硬件状态别靠软件层推断,直接读字节。"

跑完路由测试 + 拆检:

| 片段 | 层 | 备注 |
|---|---|---|
| OTP/eFuse 烧入不可逆,烧前必须 dump 基线 | **L1** | 任何不可逆烧写都对 |
| 非对称签名:私钥签镜像,公钥/其 hash 存只读区供启动时验 | **L1** | 安全启动客观原理 |
| 信任链分层(只读 root 验一级、一级验下一级)这个**概念** | **L1** | 具体分几级、谁验谁 = L2 |
| 一级签名 SOP:生成密钥→签镜像→hash 入只读区→启动早期读出验签 + 每步产物检查 | **L2** | 可迁移流程 |
| 诊断手法:验签不过 → 先字节级 dump 两端 hash 对比,别从上层现象推 | **L2** | 可迁移诊断 |
| **坑·方法层**:硬件状态禁止从软件层 flag 推断,直接读寄存器/OTP 字节 | **L2** | 一条坑拆出的可迁移教训 |
| **坑·通用类**:改了某密码参数的长度,要审整条路径上所有按旧长度定的固定缓冲区(越界冲掉相邻字段) | **L2** | 从这次实现 bug 抽出的通用教训 |
| **坑·实现层**:本 bootloader 版本的公钥缓冲区按旧密钥长度定,放更长的密钥会越界 | **L3** | 此 SoC + 此 bootloader 版本的实现事实 |
| 只读区公钥-hash 地址 / 签名算法 / PSS saltlen 值 | **L3** | 平台参数 |
| 密钥文件路径、烧录工具名 | **L3** | 平台参数 |
| 这块板当前用哪把 key、当前 HEAD | **L4** | 状态 |

**看那个缓冲区越界坑 —— 一条拆成多片**:L2 通用教训("改密码参数长度→审固定缓冲区")和方法教训("硬件状态读字节")换任何平台都救你;
L3 事实("本 bootloader 缓冲按旧长度定")只对此 SoC 此版成立。这就是 §4「拆」的威力。

## 6. 矛盾处理:不和稀泥,记成「一个 L2 模式 + 多个 L3 变体」

两个平台做法不同时(例:A 芯片把 root hash 放 OTP,B 芯片放别的只读区):
- **不要**取平均、不要二选一。
- 抽出共同的 L2 模式:"vendor 把 root-of-trust 锚在芯片内只读区"。
- 各平台的具体做法记成该模式下的 L3 变体。

矛盾本身是信号 —— 它告诉你哪部分是真不变(L2),哪部分是平台自由度(L3)。

---

## 7. 置信度 + 溯源(每条必带,缺了不收)

| 字段 | 内容 |
|---|---|
| 来源 | 哪次会话 / 哪个 commit hash |
| 时间 | 提炼日期(绝对日期,不写"当前/最近") |
| 置信度 | high / med / low |
| 复现次数 | 这条在几个平台/项目里见过 |

**L2 升级规则(对标 nuwa 跨域复现):**
- 一条教训只在 **1 个**平台见过 → 标 **low**,先放对应平台的 L3 旁注,**不进 L2 正文**。
- 同一条在 **第 2 个**平台复现 → 升 L2,置信度 med。
- 这是防"把一次性巧合当通用规律"塞进 skill 正文的关口。

---

## 8. 质量自检清单(蒸完逐条过,不过打回)

```
L1 条目
  [ ] 举不出"它在哪个平台不成立"?(举得出 → 降 L2/L3)
  [ ] 是客观约束,不是"我以为通用"的个人做法?

L2 条目
  [ ] 跨平台复现 ≥2 次?(只 1 次 → 标 low,留 L3 旁注)
  [ ] 删掉所有平台专属值后,这条还独立成立?

L3 条目
  [ ] 只放值,没夹带"为什么"?(why 属于 L2)
  [ ] 标了它属于哪个平台?

坑条目
  [ ] 已拆成「教训(L2)+ 事实(L3)」两片?(没拆 = 不合格)

通用
  [ ] 每条带 来源 + 时间 + 置信度 + 复现次数?
  [ ] 矛盾保留了(没和稀泥)?
```

## 9. 反例黑名单(提炼时绝不做,对标 darwin dim9)

| # | 反模式 | 为什么不要 | 替代 |
|---|---|---|---|
| 1 | 把具体值/地址/路径写进 SKILL.md 正文 | 换平台改不动,且每个值散落多处易漏改 | 一律降 L3,正文只引用"见 params/<平台>.md" |
| 2 | 把一条坑整条塞进一层 | 教训和事实绑死,要么没法复用要么污染正文 | 强制 §4 拆成 L2+L3 |
| 3 | 单次出现的经验直接进 L1/L2 正文 | 把巧合当规律,误导后续所有平台 | §7 复现 ≥2 次才升,否则 low + L3 旁注 |
| 4 | 拿不准就往高层放(求"显得通用") | 污染面最大,最坏的错向 | §3 不对称原则:拿不准往低放 |
| 5 | 矛盾时二选一 / 取平均 | 抹掉了"哪是不变、哪是自由度"的信号 | §6 一个 L2 模式 + 多个 L3 变体 |
| 6 | L3 参数表里写大段"为什么这么配" | 解释会随理解变,绑在数据上没法单独更新 | why 上提到 L2,params 只留值 |
| 7 | 提炼项不带溯源就入库 | 日后无法判断是否过时、来自哪次 | §7 四字段必填 |

**触发:每条提炼定稿前对照本表一次,任一命中 → 重提。**

## 10. 产物落点(提炼结果写到哪)

```
<skill 根>/
  SKILL.md            # L1 + L2:原理 + 设计 + SOP + 坑(可迁移部分),零具体值
  pitfalls.md         # L2 坑库,append-only,每条带溯源
  params/
    _template.md      # 新平台照填的空表
    <平台A>.md        # L3:该平台全部参数 + 实现细节事实
    <平台B>.md
  # L4 不在 skill 里 —— 进 memory,带时间戳
```

## 11. 更新规则(因为按作用域拆开,更新都是局部的)

| 变的是什么 | 动哪 | 不动哪 |
|---|---|---|
| 平台参数变(某密钥长度调整) | L3 一行 | L1/L2 完全不碰 |
| 方法变好(发现更优诊断) | L2 一处 | **所有平台一起受益** ← 越用越聪明的来源 |
| 新平台 | 新增一份 `params/<平台>.md` | L1/L2 不碰 |
| 状态过期 | L4 自然刷新 | —— |

旧的 L3/L4 不删,按 MemoryBank 遗忘曲线降权沉底(留溯源,可回查)。

事实类改写遵循 append + 标记替代,不静默覆盖(尤其安全/不可逆项):
旧值改了要留一行"原 X → 改为 Y,因 Z,日期"。

## 12. 一条知识 vs 一个 skill:作用域同时决定边界

分层用的尺子(作用域)不止决定**一条知识进哪层**,还决定**哪些「领域」是同一个 skill**。

**核心原则:skill 的边界 = L1/L2 的边界。**
- 两个领域 L1/L2 高度重合 → 它们是**同一个 skill 的不同 L2 切面**,合并。
- 两个领域 L1/L2 互相陌生 → **不同 skill**,分开。
- L3 的差异**不构成**新 skill —— 那只是换一张参数表。

### 判定流程

给两个领域 A、B:

```
1. 分别列出各自的 L1 / L2
2. 重合度高(共享信任链 / 原理 / 核心 SOP)
     → 合并成一个 skill;各领域降为 L2 切面 + 各自独立 L3 表
3. 重合度低(L1/L2 互相看不懂)
     → 分开成两个 skill
4. 拿不准
     → 默认分开(对照 §3 不对称:过度合并 = 正文臃肿 + description 触发不准,
        代价高于多开一个 skill)
```

### Worked example:verified-boot 家族

| 领域 | 共享 L1/L2 | 各自的 L2 切面 + L3 |
|---|---|---|
| 安全启动 | 信任链/root-of-trust、非对称签名、hash 比对、验签不过先 dump 字节 | 验在 BootROM/SPL;L3=OTP 地址 |
| AVB/vbmeta | 同上 | 验在 bootloader、vbmeta 结构;L3=vbmeta key/分区 |
| fastboot 验签 | 同上 | 验在下载流;L3=验哪些分区/时机 |
| OTA 验签 | 同上 | 验在 update_engine/recovery;L3=包签名/回滚 |
| 公钥替换 | 同上 | 是上面几个的一个操作;L3=新公钥 |

结论:这 5 个 = **1 个 `verified-boot` skill**(共享 L1/L2 当正文)+ 5 个 L2 切面 + 各平台 L3 表。
而 SDK 迁移的 L1/L2(三方对比 / 上游意图 / keep-merge-重写)跟它们互相陌生 → **独立 skill**。

### 反例(§9 黑名单的延伸)

| 反模式 | 为什么不要 | 替代 |
|---|---|---|
| 每个 task 开一个 skill | skill 爆炸,且 L1/L2 在多个 skill 里重复 → 改一处要改 N 处 = 库漂移源头 | 按 L1/L2 重合度聚类合并 |
| 把陌生领域硬塞进一个 skill | 正文臃肿、触发词冲突、召回时加载错或一次加载多个 | L1/L2 陌生就分开 |

### 和召回的关系

skill 越少、边界越准 → description 越好写 → 触发越准。**「按 L1/L2 边界聚类」直接服务召回质量**——这是 skill 库不退化的关键(SkillLens:LLM 判 skill 质量仅 46% 准,库越乱越糟)。

### 切面长大了怎么办

一个 L2 切面(如 OTA)膨胀到自成体系(大量自己的 L1/L2,跟家族其他成员开始陌生)→ 这是分裂信号,
把它拆成独立 skill。**边界不是一次定死,随 L1/L2 重合度演化**(同 §11 的更新精神)。
